# Host CSS and Table Layout

This page explains how host application CSS can affect Widgemo table rendering and how to keep behavior predictable.

## Baseline in plain apps

When an app does not apply broad global table rules, Widgemo table rendering follows widgemo-core defaults.

Expected baseline:

- Table borders, stripe behavior, and header treatment come from widgemo-core and mode config.
- Cell spacing and row density follow core defaults plus any responsive mode behavior.
- Table mode layout is mode-driven:
  - `traditional` uses fixed table layout.
  - `rich-cells` uses auto table layout.
- Runtime table sizing uses `width: max-content` with `min-width: 100%`.
- Horizontal overflow is handled by the internal scroll region (`.widgemo-table-scroll-region`).

## Row striping

`alternatingRows` (defaults `true`) is the Widgemo-native control for row striping. It applies inline `backgroundColor` to every other row using the `--widgemo-row-alt-bg` token.

Bootstrap-style hosts are supported, and with current widgemo-core behavior `alternatingRows` remains visible as the active striping contract.

The caveat is aggressive host overrides that explicitly repaint rows or cells. For example, a host rule that applies cell-level paint (`box-shadow: inset 0 0 0 9999px var(--bs-table-accent-bg)`), hard `background` values on `td`, or `tr:nth-child` repaint rules can visually override row-level striping. This is host influence, not a second Widgemo striping mechanism.

If alternating rows appear reduced or inconsistent, audit host-level `table`/`tr`/`td` repaint rules first.

## What host global table CSS can change

Many sites ship global selectors like these:

```css
/* Examples of host rules that can alter Widgemo table behavior */
table, th, td { border: 1px solid var(--brand-border); }
table tr:nth-child(even) { background: var(--striped-row); }
table th, table td { padding: 12px; }
```

These selectors can change Widgemo semantics by introducing unexpected borders, striping overrides, and spacing. In particular, `tr:nth-child` stripe rules can conflict with `alternatingRows`-driven row backgrounds.

## Docusaurus and Infima hosts only

If your host stack is Docusaurus with Infima, scoped variable resets can neutralize markdown table chrome leakage inside Widgemo showcases.

```css
/* Docusaurus/Infima-only example */
.my-app-widgemo-scope {
  --ifm-table-border-width: 0;
  --ifm-table-border-color: transparent;
  --ifm-table-background: transparent;
  --ifm-table-stripe-background: transparent;
  --ifm-table-head-background: transparent;
}
```

## Framework-agnostic scoped mitigation

For non-Infima hosts, avoid global `table`, `th`, and `td` overrides and restore Widgemo table semantics only inside a local scope.

```css
/* Example for any host framework */
.my-app-widgemo-scope .widgemo-table {
  display: table;
  margin-bottom: 0;
}

.my-app-widgemo-scope .widgemo-table-scroll-region {
  overflow-x: auto;
}

.my-app-widgemo-scope .widgemo-table > :not(caption) > * > * {
  border: 0;
}
```

Notes:

- Keep mitigations scoped to a Widgemo wrapper class.
- Do not apply these mitigations as global table rules.
- Do not add density tuning unless you intentionally want a non-default look.

## Deterministic column widths

If host CSS and content variability produce inconsistent column width distribution, define widths explicitly with `field.width`.

```tsx
fields: [
  { key: 'name', label: 'Name', width: '220px' },
  { key: 'status', label: 'Status', width: '140px' },
  { key: 'owner', label: 'Owner' },
]
```

Use `field.width` when:

- Key columns must remain stable across pages and breakpoints.
- Content length differences cause noticeable width drift.
- You need deterministic alignment with external UI elements.
