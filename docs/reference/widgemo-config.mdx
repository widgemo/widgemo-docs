# Widgemo Config

`WidgemoConfig` is the central configuration object. It is zone-oriented and mode-driven.

## Shape

| Field | Type | Default | Description |
|---|---|---|---|
| `id` | `string` | — | Optional identifier |
| `zones.header` | `ZoneConfig` | — | Optional header zone |
| `zones.content` | `ContentConfig` | **required** | Content zone (mode, fields, actions, etc.) |
| `zones.footer` | `ZoneConfig` | — | Optional footer zone |
| `interactions.onEvent` | `(ctx: InteractionContext) => void` | — | Global sink for all actions and gestures |
| `collapse.initialState` | `'expanded' \| 'collapsed' \| 'fixed'` | `'expanded'` | Widget collapse behaviour |
| `collapse.button` | `boolean` | `true` | Show/hide the collapse toggle |
| `containerFrame` | `{ shell?, border?, shadow?, borderRadius?, borderColor?, borderWidth?, overflow? }` | — | Outer shell frame controls |
| `style` | `CSSProperties` | — | Inline styles on the root element |
| `theme` | `string` | — | Registered theme name |
| `devMode` | `boolean \| DevModeConfig` | `false` | Dev inspector overlay |
| `preRender` | `() => void` | — | Config-level callback invoked by `Widgemo` before render |

## Gestures

`zones.content.gestures` defines config-driven user interaction handlers.

Supported gesture types:

- `item-click`
- `item-drag-start`
- `item-drop`

### Gesture Types

| Type | Typical Source Surface | Payload Intent | Where Handler Goes |
|---|---|---|---|
| `item-click` | Table rows, cards, and other selectable content items | User selection or activation of an item | `zones.content.gestures[].onTrigger`, otherwise `interactions.onEvent` |
| `item-drag-start` | Board cards that can be moved | Drag initiation details for a move operation | `zones.content.gestures[].onTrigger`, otherwise `interactions.onEvent` |
| `item-drop` | Board drop targets | Drop/completion details for a move operation | `zones.content.gestures[].onTrigger`, otherwise `interactions.onEvent` |

### Handler Flow

- If a matching gesture entry defines `onTrigger`, Widgemo calls that handler first.
- If `onTrigger` is not provided, Widgemo falls back to `interactions.onEvent`.
- If neither is provided, Widgemo does not run an app-level gesture handler.

```tsx
const config = {
  interactions: {
    onEvent: (ctx) => {
      console.log(ctx.kind, ctx.entity?.id);
    },
  },
  zones: {
    content: {
      mode: 'board',
      gestures: [
        {
          type: 'item-click',
          interactionId: 'open-item',
          interactionLabel: 'Open item',
          onTrigger: (ctx) => {
            openItem(ctx.entity?.id);
          },
        },
        {
          type: 'item-drag-start',
          interactionId: 'move-start',
          interactionLabel: 'Start move',
        },
        {
          type: 'item-drop',
          interactionId: 'move-drop',
          interactionLabel: 'Finish move',
        },
      ],
      item: { fields: [{ key: 'name' }], layout: { type: 'auto' } },
    },
  },
};
```

For a quick decision guide on gestures/interactions vs lifecycle hooks, see [Extension API: When to Use Which](/core/reference/extension-api#when-to-use-which).

### `interactions.onEvent`

`interactions.onEvent` is the global interaction sink for actions and gestures when no local handler is provided.

For the canonical lifecycle hook contract, see [Extension API](/core/reference/extension-api#lifecycle-hooks).

## Lifecycle Hook Entry Points

`WidgemoConfig.preRender` is a config callback entry point.

- It is called as `() => void` by `Widgemo` before render.
- It is separate from registry lifecycle hooks (`registerWidgemoHook`).

`zones.content.gestures` and `interactions.onEvent` are interaction handlers, not lifecycle hooks.

For the canonical lifecycle hook contract (`preRender`, `postRender`, `onItemClick`, `onModeChange`, `onDragDrop`), see [Extension API](/core/reference/extension-api#lifecycle-hooks).

For a quick decision guide on gestures vs lifecycle hooks, see [Extension API: When to Use Which](/core/reference/extension-api#when-to-use-which).

## Header and Footer Zones

```tsx
zones: {
  header: {
    title: 'Team Directory',
    subtitle: 'All active employees',
    icon: { name: 'users', size: 28, color: '#4f46e5' },
    // Dynamic title driven by live data
    titleRenderer:    (data) => `Team (${data.length} members)`,
    subtitleRenderer: (data) => `Active: ${data.filter(d => d.status === 'active').length}`,
    layout: {
      titlePosition:    'center',    // 'left' (default) | 'center' | 'right'
      iconPosition:     'above',     // 'left' (default) | 'above'
      actionsPosition:  'below',     // 'right' (default) | 'below'
      subtitlePosition: 'inline',    // 'below' (default) | 'inline'
      orientation:      'vertical',  // 'horizontal' (default) | 'vertical'
    },
    actions: [ /* ActionConfig[] */ ],
    themeOverrides: { /* ZoneTheme */ },
    style: { background: 'linear-gradient(…)' },
    collapse: { initialState: 'collapsed', button: true },
    enabled: true,
  },
  footer: {
    subtitle: 'Last synced just now',
    layout: { titlePosition: 'center' },
  },
},
```

## Container Frame

`containerFrame` is the canonical first-release shell control.

Supported fields:

- `shell`: `'on' | 'none'` — **new** — controls whether the outer container surface renders at all (see [Shellless mode](#shellless-mode-shell-none) below)
- `border`: `'none' | 'on'` — shell border line
- `shadow`: `'none' | 'on'` — shell drop shadow
- `borderRadius`: `'square' | 'rounded' | number | string`
- `borderColor`: `string`
- `borderWidth`: `string | number`
- `overflow`: any valid CSS `overflow` value

Default for all fields: `undefined` (core defaults apply — bordered, with shadow, with background).

```tsx
const config = {
  containerFrame: {
    border: 'on',
    shadow: 'none',
    borderRadius: 10,
    borderColor: '#d0d7de',
    borderWidth: 2,
    overflow: 'hidden',
  },
  zones: { /* … */ },
};
```

### Shellless mode (`shell: 'none'`)

When `shell: 'none'`, the outer shell surface is disabled while zone internals continue to render normally:

- Outer background is transparent
- Border and shadow are removed
- Header and footer default to transparent backgrounds
- Outer radius defaults to `0` unless explicitly configured
- Outer overflow defaults to `visible` unless explicitly configured
- `zones.content` spacing remains mode-owned

Shellless affects the outer container only; mode-level and zone-level spacing behavior is unchanged.

```tsx
const config = {
  // The widget has no visible card surface.
  // It floats inside the host surface with normal zone spacing.
  containerFrame: { shell: 'none' },
  zones: {
    header: { title: 'Approvals', subtitle: 'Inline queue' },
    content: { mode: 'table', item: { fields: […], layout: { type: 'auto' } } },
  },
};
```

### Borderless vs shellless — when to use which

| | Borderless (`border: 'none', shadow: 'none'`) | Shellless (`shell: 'none'`) |
|---|---|---|
| **Outer card background** | Present (uses theme background) | Transparent |
| **Border line** | Removed | Removed |
| **Drop shadow** | Removed (when `shadow: 'none'`) | Removed |
| **Header/footer default background** | Present (theme/card surfaces) | Transparent by default |
| **Outer radius default** | Theme/card default radius | `0` (unless explicitly configured) |
| **Outer overflow default** | Theme/container default behavior | `visible` (unless explicitly configured) |
| **Zone padding** | Normal | Normal |
| **Visual effect** | Card-like block, chrome stripped | Blends into host surface |
| **Use when** | You want a flat card that fits inside a panel but keeps its own background | You want the widget to read as part of the page, with no card boundary at all |

**Key rule:** if you can still see the widget's white/dark card rectangle after removing the border and shadow, you have borderless. If that rectangle is gone and the widget content floats on the host surface, you have shellless.

### Validation Checklist

- In borderless shells (`containerFrame.border: 'none'`), confirm header/footer spacing remains consistent with bordered shells unless you explicitly override zone padding.
- In shellless widgets (`containerFrame.shell: 'none'`), confirm host background shows through the widget area.
- In shellless widgets, confirm header/footer do not leave behind card-like background surfaces by default.
- In shellless widgets, confirm no rounded outer card boundary remains unless radius is explicitly configured.

## Loading and Error States

Loading and error behavior is runtime-driven by `zones.content.status` plus the matching state object.

### Loading indicators (built-in runtime behavior)

- `spinner`: renders the built-in spinner indicator.
- `skeleton`: renders the built-in skeleton placeholder bars.
- `text`: renders the built-in text indicator (`Loading`).

`loadingState.indicator` selects built-in loading presentation only.
`loadingState.renderer` is the only custom-loading override path.

```tsx
content: {
  mode: 'table',
  status: 'loading',
  loadingState: {
    enabled: true,
    indicator: 'skeleton',
    message: () => 'Loading records…',
  },
  item: { fields: […], layout: { type: 'auto' } },
},
```

```tsx
content: {
  mode: 'table',
  status: 'loading',
  loadingState: {
    enabled: true,
    indicator: 'spinner',
    message: 'Loading records…',
    renderer: ({ message, indicator, data }) => (
      <div className="my-loading-shell">
        <strong>{message}</strong>
        <span>indicator={indicator}</span>
        <span>rows={data?.length ?? 0}</span>
      </div>
    ),
  },
  item: { fields: […], layout: { type: 'auto' } },
},
```

### Error severity (built-in runtime behavior)

- `info`: uses built-in info tone/styling.
- `warning`: uses built-in warning tone/styling.
- `error`: uses built-in error tone/styling (default).

Severity affects built-in tone/styling only. It does not change retry wiring, error message resolution, or data flow.

```tsx
content: {
  mode: 'table',
  status: 'error',
  error: { message: 'Failed to load data' },
  errorState: {
    enabled: true,
    message: (err) => `Error: ${err?.message ?? 'Something went wrong'}`,
    retry: { label: 'Try Again', onRetry: () => refetch() },
    severity: 'warning',
  },
  item: { fields: […], layout: { type: 'auto' } },
},
```

```tsx
content: {
  mode: 'table',
  status: 'error',
  error: { message: 'Failed to load data' },
  errorState: {
    enabled: true,
    severity: 'info',
    retry: { label: 'Try Again', onRetry: () => refetch() },
    renderer: ({ message, severity, showRetry, retryLabel, onRetry }) => (
      <div className={`my-error my-error-${severity ?? 'error'}`}>
        <p>{message}</p>
        {showRetry ? <button onClick={onRetry}>{retryLabel}</button> : null}
      </div>
    ),
  },
  item: { fields: […], layout: { type: 'auto' } },
},
```

### Renderer override precedence

- `loadingState.renderer` overrides built-in loading indicator UI (`spinner`, `skeleton`, `text`).
- `errorState.renderer` overrides built-in severity UI (`info`, `warning`, `error`).

When a renderer override is provided, built-in visuals are bypassed and your renderer output is rendered instead.

Retry callback note:

- `zones.content.errorState.retry` controls content-level retry UI.
- For externally managed error flow (`<Widgemo error={...} onRetry={...} />`), top-level `onRetry` is used as the retry callback for the content error action. If `zones.content.errorState.retry` is an object, existing fields such as `label` are retained while `retry.onRetry` is overridden.
- Current external-flow nuance: if `zones.content.errorState.retry` is `false`, retry remains disabled even when top-level `onRetry` is provided with `error`.
- Top-level `onRetry` also powers retry in the runtime boundary fallback UI.

When `errorState.renderer` is used, it receives the resolved retry-related props from runtime:

- `showRetry`: whether retry should be displayed.
- `retryLabel`: resolved retry label.
- `onRetry` and `retry`: callback aliases for backward compatibility.
